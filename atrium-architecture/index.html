System design · public walkthrough

A shared, durable, versioned workspace for a fleet of agents

How dozens of AI coding agents — and the humans supervising them — can share lots of files and reconcile their edits, when every agent runs locked inside a sandbox that nothing is allowed to connect into.

Atrium agent platform Postgres + S3 backing No-ingress sandboxes Updated June 2026

01The problem

Many agents, lots of shared files, and a hard isolation constraint.

Atrium runs teams of AI agents working in parallel under human oversight. A realistic session isn't one agent in one repo — it's a dozen agents and several people, all touching an overlapping pool of documents, datasets, notebooks, and code. That creates four requirements that pull against each other:

  • Durability — an agent's work must survive a crash, a pause, or the sandbox being destroyed.
  • Freshness — when a teammate (human or agent) changes a shared file, others should see it quickly, not discover it hours later.
  • Conflict-safety — if two actors edit the same file at once, neither edit may be silently thrown away.
  • Agent UX — agents should just write files. No "remember to commit," no version ceremony. The plumbing happens underneath them.
The constraint that shapes everything Each agent runs in a hardened, “no-ingress” sandbox: it is non-privileged, drops all Linux capabilities, and — critically — nothing can open a connection into it. The sandbox can only reach out. That single rule rules out every off-the-shelf file-sync tool (Syncthing, Mutagen, rsync daemons, webhooks) because they all need to connect in. So all synchronization has to be either sandbox-initiated or driven from outside the sandbox entirely.

02Three data shapes, three models

The first design decision: don't force everything into one consistency model.

Agent work produces three kinds of data, and they want fundamentally different handling. Trying to unify them is the classic mistake.

ShapeExamplesBackingMerge model
Logsconversation transcripts, the agent's step-by-step tracePostgres event log → sealed S3 segmentsAppend-only — never merged
Artifactsdocuments, datasets, reports, notebooks, imagesContent-addressed blobs (S3) + version ledger (Postgres)Per-file merge — versioned, conflict-aware
Codesource repositoriesGit, on the existing forgeGit — we coordinate around it, we don't reinvent it

The interesting one is the middle row. Logs are easy (just append). Code is solved (git already exists, and agents already know it). Artifacts are where the real work is: files that both humans and agents edit, that need history, that need to be shared across sessions, and that need a sane answer when two edits collide. The rest of this page is mostly about that layer.

03The content-addressed ledger

Git's data model, rebuilt on S3 + Postgres, with one twist for agents.

The artifact store borrows git's core idea: immutable content + moveable pointers. Every version of every file is stored once, addressed by the hash of its bytes (so identical content is automatically de-duplicated). An "edit" never overwrites anything — it writes a new blob and advances a pointer. Old versions are always retrievable.

LayerLives inWhy
Immutable file bytesS3, keyed cas/<sha256>Object stores are cheap and infinite, but can't do atomic “compare-and-set”
Version chain, pointers, conflict statePostgresA transactional database can do atomic updates — so the moveable parts live here

The twist: conflicts are state, not errors

Most version systems, when two edits collide, either block ("merge failed, fix it before you can continue") or silently pick a winner (last-write-wins — the other edit is gone). Both are wrong for autonomous agents: blocking stalls a fleet, and silent loss is unacceptable.

Atrium follows the model pioneered by the Jujutsu (jj) version control system: a conflict is a first-class, recorded version that contains both sides. The pointer still advances — nothing blocks — but the new “latest” version is explicitly marked conflicted and carries both competing edits. A human or agent resolves it later, as a normal follow-up edit. Nothing is ever lost, and nothing ever stalls.

Why build this instead of buying it We evaluated the obvious off-the-shelf option (lakeFS, “git for data”) by actually running it. It only does whole-file merges — a collision returns an error or silently discards a side — and its per-tenant access control is paywalled. The conflict-as-state model and tenant isolation we need are exactly what it can't give, so we'd be rebuilding them on top anyway. jj has the right model but no open-source cloud backend exists (the scalable one is Google-internal). So the ledger is our own — but it's small: content-addressed blobs and a version chain are well-understood, and the bytes plumbing reuses infrastructure Atrium already runs.

04The agent's workspace

A copy-on-write filesystem, so the agent writes freely and the change-set is free.

Inside the sandbox, the agent sees a normal directory at /workspace. Underneath, it's an overlay filesystem with three layers:

/workspace  — what the agent sees & edits
upper  — this agent's changes (writable, private)
lower  — hydrated shared files + deps (read-only)
A write copies the file up into the private upper — the shared base is never touched.

This buys two things for free. First, the agent gets full POSIX behaviour — real renames, locks, partial writes — with no “permission denied” errors, because every write lands in its own private layer. Second, and crucially, the upper layer is the exact set of changes the agent made. There's no need to scan the whole tree to find what changed; the filesystem already separated it out.

A deliberate scoping trick The overlay covers only the shared-artifact folders. Big, noisy directories — node_modules, virtualenvs, build caches, and git repos — are mounted as separate volumes. That means they structurally can't bloat the change-set or get captured by accident, no filtering rules required. A dependency install simply isn't in the layer we watch.

05Capture — getting changes out

A scan that runs outside the sandbox, so the agent stays fully locked down.

Here's the tension: we want to watch the agent's changes and ship them to durable storage — but the sandbox is hardened and can't run privileged watchers, and (per the constraint) nothing can connect into it to read them out.

The answer is to move the work out of the agent and onto the machine. A privileged node daemon — one per physical host, shared by all the agents on it — reads each agent's upper layer directly from the host side:

upper layer
only the changed files
→
node daemon
reads, interprets, hashes
→
S3 + ledger
new version recorded
The daemon reads from the host; the agent never runs a watcher and never opens an inbound port.

Because it reads the change-set rather than the whole tree, the work is proportional to what actually changed, not the size of the workspace. And because there's one daemon per machine rather than one watcher per agent, it scales with the number of hosts, not the number of agents. The daemon also understands the filesystem's encoding for deletes and renames, so those are captured faithfully — and since it's trusted infrastructure, it can stream large files straight to S3.

Verified, not assumed The load-bearing question — “can a fully locked-down agent's writes, through an overlay it doesn't control, be read by a node component?” — was proven end-to-end on a real Linux kernel and a real Kubernetes test cluster before committing to the design.

06Sync — getting changes in

Freshness without ever connecting into the sandbox.

The hard direction is inbound. If a teammate edits a shared file, a running agent is still looking at the version it started with — its read-only lower layer is frozen at startup. Left alone, the agent does work on stale data and never finds out. (Our analysis of a busy shared folder found this “acted on stale data, no signal” case is the quiet, common danger — more so than outright collisions.)

We can't push into the sandbox, so the flow splits across the boundary:

StepRuns whereWhat happens
Fetch & stagenode (outside)The daemon notices the ledger advanced a file this agent is using, fetches the new bytes, and drops them in a side /incoming folder with a marker.
Notifynode → agentA lightweight signal: “these paths changed.” The bytes never travel over the agent's input channel — just a pointer.
Adoptinside, at a safe pointThe agent merges the staged version into its live workspace between steps — never mid-write.

Adoption handles three cases: if the agent hadn't touched the file, the new version simply replaces it; if the agent had edited it too, a three-way merge runs (clean → merged, otherwise → a recorded conflict); if the agent deleted it but someone else edited it, the file is resurrected. The target is that a running agent sees a teammate's change within seconds, and an autonomous agent rebases onto it at its next natural checkpoint rather than being interrupted mid-thought.

07Durability by content class

One capture mechanism, routed to the right home for each kind of data.

The node daemon is a single mechanism, but it doesn't force everything into the version ledger — that would be wrong for logs (which churn) and for code (which has its own branch model). It routes by content class:

ContentGoes toWhy there
Artifactsthe version ledgeredited, versioned, shared — the conflict-aware store
Uncommitted code editsa patch/diff snapshot (no git refs)keeps work-in-progress durable without touching the repo agents share — see below
Logs & transcriptspacked, sealed S3 segmentsappend-only; needed to resume a session, not to version
Design decision: how to keep uncommitted code safe Uncommitted edits in a code repo also need to survive a crash. The tempting approach — quietly auto-committing them to a hidden git ref — was rejected: any automated git activity in a repository a fleet of agents shares risks confusing those agents (unexpected objects, refs, and garbage-collection churn). Instead, the daemon captures work-in-progress as a plain patch/diff snapshot stored as an artifact. It creates zero git objects or refs the agent can see, so it can't surprise anyone. The trade-off: recovery means “re-clone and apply this patch,” rather than a native git checkout — an acceptable price for keeping the agent's view of git pristine.

08Build vs buy

Buy the algorithm, borrow the pattern, build the glue.

A lot of this resembles problems open-source infrastructure already solved — chunking, deduplication, log storage, filesystems over object stores. We deliberately checked each layer (five parallel research passes, hand-computed economics, and a stress-test against real user stories) to avoid reinventing commodity infrastructure. The split that fell out:

Buy

Mature libraries we adopt directly: a content-defined chunker (fastcdc-rs) and Hugging Face's now-open Xet de-dup layer for very large files; Vector to ship logs; Parquet for cold archives.

Borrow

Well-known patterns we re-implement in a few hundred lines, because adopting the whole product would be overkill: the log-segment + index shape, and the patch-snapshot idea for work-in-progress.

Build

The parts no product offers: the conflict-state ledger, the no-ingress sync, the node-side capture, hydrating the read-only base, and the agent-facing policy (scoping, merge rules, filtering).

The headline finding No off-the-shelf system does this combination — no-ingress, large binaries, versioned-with-merge, multiple data shapes, agent semantics. We surveyed how the leading agent platforms handle it, and the universal answer is “ephemeral sandbox per task + git pull requests” — nobody ships a durable, shared, conflict-aware workspace store. The few products in the neighbourhood are either closed competitors or plain last-write-wins volumes. The integration is the product.

09Scaling to a team or org

What changes when it's 50–200 agents instead of one.

The design was sized for fleet scale, and a few numbers drove the choices:

  • Don't make one storage object per event. A naive “one object per log line” approach would cost thousands of dollars a month in request fees and create tens of millions of tiny objects a day. Batching log lines into sealed segments before they hit object storage is ~120–300× cheaper and is the difference between tractable and not.
  • Content-addressing spreads load for free. Because files are keyed by their hash, writes naturally fan out across hundreds of storage partitions, so the object store's per-partition write limits are a non-issue.
  • Conflicts are real at scale. On a hot shared folder with dozens of active agents, a meaningful fraction of writes overlap with another in-flight edit. That's exactly why conflict-as-state and fast inbound sync aren't over-engineering — they're load-bearing once sharing is real.
  • De-duplication of large files is deferred on purpose. Chunk-level de-dup saves 14–34× on big files edited in small increments, but almost nothing on ordinary text. So it's switched on only when large, churny binaries actually dominate storage — and even then we adopt a library rather than write one.

Tenant isolation

The node daemon can read every agent's files on its machine — fine within one team, a real boundary between separate organizations. The plan for true multi-tenancy is a VM per tenant: each tenant's agents run on their own virtual machine, so the “can read everything on the host” power is contained by a hardware boundary rather than a software policy. (This is designed but not yet built — see below.)

10Limitations & open questions

An honest accounting of what isn't solved.

This is a design with a working core, not a finished system. The status, roughly:

Built: the conflict-state ledger (content-addressed blobs, version chain, conflict storage, write-back, garbage collection)  ·  Proven but not productionized: the node-side capture mechanism  ·  Designed, not yet built: inbound sync, base-layer hydration, multi-tenant VMs

Known limitations

  • Stale reads have a window. Fast inbound sync narrows it to seconds, but an agent can still act briefly on data that was superseded between its last sync and its next one. We make the window small and visible; we don't eliminate it.
  • Large files have a ceiling on the in-sandbox path. The node daemon can stream big files out, but the sandbox-initiated path is capped (~16 MiB) until the streaming + chunking layer lands. Fine for documents and small datasets; not yet for multi-gigabyte media.
  • Some conflict resolutions are a product call, not an algorithm. The clearest example: if one agent deletes a file while another edits it, should it stay deleted or be resurrected? There's no universally correct answer — it needs a deliberate UX decision.
  • Resume depends on a stable base. When a paused agent comes back, its private edits are re-stacked on a freshly rebuilt read-only base. That base must be byte-identical to the original, or the edits drift. The version manifest makes this tractable, but it's unverified on real hardware.
  • The work-in-progress patch isn't a perfect git mirror. The patch-snapshot approach (chosen to keep the shared repo pristine) doesn't faithfully capture every exotic git state — an in-progress rebase, say. It's a recovery point, not a clone.
  • Multi-tenant isolation is designed, not deployed. The VM-per-tenant boundary is the plan; today's single-tenant use doesn't exercise it, and the virtualization density work is future.
  • A privileged node component is a serious surface. It runs with host privileges and reads many agents' files — which is exactly why the tenant boundary has to be hardware, and why it gets hardened and audited as trusted infrastructure.

Where we're behind the field — on purpose

Competitors ship a couple of polished workflows we have the primitives for but haven't surfaced: instant whole-environment forking (“branch this entire agent's machine in a fraction of a second to try an alternative”), and a first-class “run several agents, keep the best result” loop. These are UX layers on top of the storage model described here — natural next steps rather than missing foundations.

The one-sentence summary Give every agent a normal local filesystem to work in; capture and reconcile everything from outside the sandbox through a content-addressed, conflict-aware ledger — so a fleet of agents and their humans can share a workspace without a central server they're allowed to connect to, and without ever silently losing an edit.
